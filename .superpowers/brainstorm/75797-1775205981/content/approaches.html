AI Briefing Architecture — 3 Approaches

How should the AI generation layer integrate with your existing pipeline?

A

Direct Replace — New ai-briefing.ts

Create a single new module that takes registry.json as context, calls Claude Haiku, and returns the Telegram message string.

scanner → scoring → registry.json → ai-briefing.ts (Claude API) → telegram
scanner → scoring → briefing.ts → BRIEFING.md (unchanged)

Pros

  • Minimal changes — one new file, one swap in the notify pipeline
  • registry.json already has everything the AI needs
  • Easy to test independently

Cons

  • System prompt lives as a string in code — harder to iterate on
B

Prompt Template — Separate prompt file + module

Same as A, but the system prompt lives in a .md file (e.g., prompts/briefing.md) that gets loaded at runtime. The module handles API call + context injection.

scanner → scoring → registry.json → ai-briefing.ts + prompts/briefing.md → telegram

Pros

  • Prompt is easy to edit and iterate on without touching code
  • Could version/swap prompts for different briefing styles
  • Same minimal code changes as A

Cons

  • Extra file to manage — slightly more moving parts
C

AI Service Layer — Generic Claude client + briefing use-case

Build a reusable ai.ts client module (handles API key, model config, retries), then ai-briefing.ts uses it for the briefing use-case. Ready for future AI features.

scanner → scoring → registry.json → ai-briefing.ts → ai.ts (Claude client) → telegram

Pros

  • Clean separation — API concerns vs briefing logic
  • Ready if you later add conversational features

Cons

  • More abstraction than needed for a single use case right now
  • YAGNI — premature if you stick with option 1 (AI briefings only)

Recommendation: Approach B — best balance of simplicity and iterability. The prompt is the thing you'll tweak most, so keeping it in its own file pays off immediately.
